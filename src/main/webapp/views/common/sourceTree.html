👥 組員操作流程

第一步：接受邀請

📧 檢查 Email 邀請信
組長發送邀請後，你會收到來自 GitHub/GitLab 的邀請信件。點擊信件中的「Accept invitation」按鈕即可加入專案。

第二步：複製專案到本地

⚡ 快速步驟：
  1. 開啟 SourceTree
  2. 點選「Clone」按鈕
  3. 貼上專案連結（向組長索取）
  4. 選擇本地存放位置
  5. 點選「Clone」完成

好的 Commit 訊息範例

✅ 好的例子： feat(登入): 新增使用者登入功能 實作使用者登入系統，包含： - 登入表單驗證 - 密碼加密處理 - 登入狀態管理 ❌ 不好的例子： 修改 update test fdasfdasf

🚨 常見問題處理

問題 1：推送時出現衝突

⚠️ 錯誤訊息：
"Updates were rejected because the remote contains work that you do not have locally."

衝突標記範例：

<<<<<<< HEAD function login(username, password) { // 你的修改 return authenticateUser(username, password); } ======= function login(user, pass) { // 別人的修改 return validateAndLogin(user, pass); } >>>>>>> commit_hash
🔧 解決步驟：
  1. 先執行 Pull 拉取最新版本
  2. 開啟衝突檔案
  3. 手動決定保留哪些內容
  4. 刪除衝突標記（<<<, ===, >>>）
  5. 儲存檔案
  6. Commit 合併結果
  7. Push 到遠端

問題 2：不小心提交錯誤檔案

💡 最佳實踐建議

✅ 應該做的事

  • 🔄 每次工作前先 Pull
  • 📝 寫清楚的 Commit 訊息
  • 🧪 提交前先測試程式碼
  • 💬 修改別人程式碼前先溝通
  • 📁 使用 .gitignore 忽略不必要檔案

❌ 絕對不要做

  • 🚫 直接修改他人程式碼
  • 🐛 提交未測試的程式碼
  • 💥 使用 git push --force
  • 🔑 提交密碼或 API 金鑰
  • 📝 使用無意義的 commit 訊息

檔案管理建議

✅ 應該提交到 Git： - 原始碼檔案 (.py, .js, .html, .css 等) - 設定檔案 (config.json, requirements.txt 等) - 文件檔案 (.md, .txt 等) - 專案相關的小型資源檔案 ❌ 不應該提交到 Git： - 編譯後的檔案 (.exe, .class 等) - 臨時檔案 (.tmp, .log 等) - 個人設定檔案 (.vscode, .idea 等) - 大型媒體檔案 (影片、大圖片等) - 敏感資訊 (密碼、金鑰等)

🎯 記住這三個黃金法則

🔄
工作前先 Pull
避免版本衝突
💬
清楚的 Commit 訊息
方便團隊協作
🧪
測試後再 Commit
確保程式碼品質
